DC AI API Proxy Documentation

Welcome to the official developer documentation for the DC AI API Proxy. This proxy supports native protocol translation, load balancing, cooldown rotations, and exposes clean compatibility layers for OpenAI, Gemini, and Claude formats.

Key Architecture Advantages

Instead of locking integrations into a single LLM provider, this proxy allows client configurations to reference a unified base URL and dynamically query backend models using standard APIs.

Stateless Load Balancing

API keys configured by the admin are rotated on a stateless Round-Robin strategy based on database usage metrics. Key cooldown periods (30s) are automatically applied on HTTP 429 errors.

Base URL Endpoints

Configure your client SDKs or integrations to target one of the following proxy roots:

Client Integration Proxy Base URL Target
OpenAI Compatible http://<domain>/v1 or http://<domain>/openai/v1
Gemini Native SDK http://<domain>/v1beta
Claude Native API http://<domain>/v1

Authentication Guide

All requests to the DC AI API Proxy must be authenticated. The proxy authenticates requests using Client Keys generated from the Admin dashboard panel.

Supported Authentication Headers

Depending on the protocol format requested, the client key can be passed in different locations:

Protocol Format Header / Query Key Value Format
OpenAI Style Authorization Bearer dc_xxxxxx...
Claude Style x-api-key dc_xxxxxx...
Gemini Native (Header) x-goog-api-key dc_xxxxxx...
Gemini Native (Query) URL parameter: ?key= dc_xxxxxx...

Security Best Practice

Do not expose client keys directly in browser applications. Use middleware or secure environment variables to hide the credentials from public networks.

Merged Models Catalog

Get a unique merged catalog of all selected models configured on all active upstream keys.

GET /v1/models

Description

This endpoint returns the unique union of all selected models configured on all active upstream keys. If no active keys exist, the default fallback model configured in the settings is returned.

Headers

Header Type Description
Authorization String Required. Format: Bearer dc_...

Example Response (JSON)

HTTP 200 OK 
{
  "object": "list",
  "data": [
    {
      "id": "gemini-2.5-flash",
      "object": "model",
      "created": 1677610602,
      "owned_by": "proxy"
    },
    {
      "id": "claude-3-5-sonnet-latest",
      "object": "model",
      "created": 1677610602,
      "owned_by": "proxy"
    }
  ]
}

OpenAI Chat Completions

Standard OpenAI-compatible completions endpoint. The proxy parses model fields and routes requests accordingly.

POST /v1/chat/completions

Description

Creates a completion response for chat messages. Automatically cleans headers, checks active model mappings, and executes automatic retries on key failure.

Request Schema (Parameters)

Parameter Type Required Description
model string Yes The target model identifier (e.g. gemini-2.5-flash)
messages array Yes Array of chat message objects (role & content)
stream boolean No Enables SSE streaming if set to true. Default: false

Example Request (JSON)

Request Body 
{
  "model": "gemini-2.5-flash",
  "messages": [
    {
      "role": "user",
      "content": "Hello, how does key rotation work?"
    }
  ],
  "stream": false
}

Claude Native Messages

Anthropic-compatible Claude native message generation endpoint.

POST /v1/messages

Description

Provides a native route for Claude requests. The proxy injects the required version headers (anthropic-version: 2023-06-01) and removes conflicting params (e.g. top_p combined with temperature) automatically.

Headers Required

Header Type Description
x-api-key String Required. Client proxy API Key.

Example Request (JSON)

Request Body 
{
  "model": "claude-3-5-sonnet-latest",
  "max_tokens": 1024,
  "messages": [
    {
      "role": "user",
      "content": "Hello Claude!"
    }
  ]
}

Gemini Native API

Standard Google Gemini API native route for content generation.

POST /v1beta/models/{model}:generateContent

Description

Supports Google's native Gemini payload schema. The proxy automatically cleans up strict validation properties from JSON bodies (flattening nested anyOf, oneOf, allOf schemas; simplifying union types; stripping enumDescriptions) to prevent upstream HTTP 400 validation issues.

Example Request (JSON)

Request Body 
{
  "contents": [
    {
      "parts": [
        {
          "text": "Write a hello world script in Go."
        }
      ]
    }
  ]
}
